DESCRIPTION           ^Top

This module provides a few test related subroutines, which can record and display test results and logs via indicated channels.

QUICK START GUIDE            ^Top

It¡¯s easy to use the subroutines provided by TestRecorder package, just like using any general package for Perl. You need install the TestRecorder Package first, and then, in your code, use or require this module, invoke any subroutines provided by it. Comparing general packages, the TestRecorder package might be much easier. This module defines a bunch of public constants, subroutines and different ways of test records. All the public constants and subroutines are exported by the use Pet::Recorder::TestRecorder; statement.

While you are working in the Sycamore SQA Automation environment, that means you are running your code in a view of sqa VOB, this package is already installed, so that you can directly use the package without any worry about the installation. For more detailed information about the public members of this module, refer to CONSTANTS, SUBROUTINES and RECORD OPTIONS parts.

From reviewability and maintainability point of view, we suggest all scripts writers to follow the conventions below while writing scripts:

-         Invoke TestSuite related subroutines to indicate the attributes of the test in TestSuite.pl file or TestPlan.pl file.

-         Invoke TestStart subroutine to start test, and indicate the test attributes as arguments for it.

-         Start a test case by using the CaseStart() subroutine, followed by a bare block.

-         Finish a test case by using the CaseEnd() subroutine, after block of the case.

-         Multiple TestStep related subroutines can be invoked in a case block, each invoking means a test step to be recorded.

-         Use Log related subroutines to record any logs, which will be recorded in log file.

-         Invoke TestEnd subroutine to finish a test task.

-         Use Environment Variables to define the test report path and formats of test records if you do not like to default setting.

Steps(StepsStates: String):

To run Tea-like steps or pure Perl statements. The parameter is a multi-line string. Each line stands for a test step or a Loop/Endloop indicator. The format of each step could be TEA-like step format, and could also be purl Perl statement.

In a TEA format test case file, for each case, each line of the case will be considered as one step of Steps defined here, so the description below in this section is also applied for TEA case writing.

Step Format:

    The following format for steps is supported:

    [[[symbol]::method]::<arg>::¡­]<arg>[<operation><expectation>][;#PassDesc[#FailDesc]]

symbol

The alias to a module defined in the test bed file

method

A valid public method of the module represented by symbol

arg1..argn

An argument to be passed to the method, you can pass up to 10 arguments to a method

operation

If you would like to do something with the data returned by executing <symbol>::<method>::<arg1> other than check the success, you may compare the result against <expectation> using one of the following operations (<,>,==,!=,=~,!~)

expectation

The expected result of the command

Once there is a ¡°::¡± presenting in the step, it stands for a Tea-like step. In a Tea-like step, there is no ¡®$¡¯ symbol required for an object. When there is no ¡°::¡± presenting in the step, it stands for a Perl statement, you can just follow Perl programa.

Variables and Expressions:

It is allowed to use variables and expressions in each step. Since all steps will be considered as a multi-line string, you need to be careful about the variables and expressions will be passed to as parts of the argument. There are two options to pass the argument, one is using double quotation marked string, and another is single quotation marked string. Both are valid, however, you need to be sure that all variables will be replaced by their values before Steps function gets the argument, if it is a double quotation marked string, while it will not happen if the argument is a single quotation marked.

E.g.:

                        my ($obj, $method, $value) = (¡®Port¡¯, ¡®get_alarm¡¯, ¡®LOF¡¯);

                        Steps (¡°$obj::$method=~$value¡±); ## same as Steps(¡°Port::get_alarm=~LOF¡±);

                        Steps(¡®$obj::$method=~$value¡¯); ## different from the above one.

In Steps function, both of above statements are valid. The difference is that

                        In a double-quotation marked string, You can use any valid variables in the lexical context of the current Perl program; while

                       In a single-quotation marked string, you have to use valid variables in main name space, i.e. they are not limited in any defined package, or a lexical block. 

It is suggested that you defined all variables which might be used in Steps() in main name space by using our ¡­ statements.

 

On Step Fail

Since the argument is a string. There will be no any compilation error. All errors will happen in running time if any. The Steps() function will parse each step, once it can not be executed, or there is any syntax error, the step will fail with the step description, and there will be a Error log before the step result to show the fail reason.

Example Command (step):

Port::alarm_present::LOS (a Boolean operation, success on a true return value, similar as Step(Port->alarm_present(LOS)) )

Port::get_alarms=~Loss of Signal (success on a return value matching text, similar as Like(Port->get_alarm, ¡®Loss of Signal¡¯) )

::open::$fh::myfilepath   (If no symbol presented, invoke a public function, similar as Step(open($fh, myfilepath)) )

::::$temp::$result=~OK    (calculate $temp and $result, success on $result matching text (OK)  similar as { $tmpresult=$temp, $result; Like($tmpresult, OK);} )

$Card1->reboot(soft); #Reboot Card Succeed # Reboot Card Failed  (If no :: presented, treat as a Perl scripts, run it, and recorder, similar as Step($Card1->reboot(soft), ¡°Reboot Card Succeed¡±, ¡°Reboot Card Failed¡±) )

 

Loop Support (This just works in Steps function in Perl, not in TEA format)

One line in Steps parameter could be a Loop/Endloop indicator. A pair of Loop/Endloop means the steps in the loop will be run multiple times which indicated in Loop line. E.g.

Loop 5

    Port1::Set::AdminDown

    Port1::Set::AdminUP

Endloop

The above loop will be executed 5 times. Loop can also be nested, that means multiple level loops can be used in one Steps function.

Endloop indication can be omitted, when the endloop is after the last step of the Steps.

Note: This loop function just works in Steps function, not in TEA format case. In TEA format case, the for statement can be used for loop. E.g.

    for (1..5) {              

                    Port1::Set::AdminDown

                    Port1::Set::AdminUP

    }

Though all test results will be recorded case by case, step by step, no matter each of them passed of failed. However, once a step/case fails, extra options or diagnostics can be helpful for test engineer to get more information and control the test.

RECORD OPTIONS          ^Top

TestRecorder is a powerful module, which provides different ways (via different channels) to save test reports. You can just work with the default setting for easy use, or you can change the setting by set Environment Variables in your operation system, or even you can set it in your scripts by invoking record option subroutines.

 

In general, there are 3 reports to be recorded via different report channels. They are SummaryReport, DetailedReport, and TestLog.

-         SummaryReport: record all test case pass/fail information, and do statistics of all the test plan;

-         DetailedReport: record all test cases and all test steps in these cases pass/fail information; and also statistics

-         TestLog: record all report information, including test cases/steps runtime info, error/warning/info/debug logs

 

However, not every kind of report defined as recorder channels will record all the 3 reports. The following table shows the detail of supports in this version:

V: supported in this release;                n/s: not supported in this version

 

If the test recorder does not support one kind of report in a specific channel, then it will do nothing about this kind of report on that channel even though both of the channel and report are enabled. E.g. in the current version, XML is not supported, i.e. no any XML reports will be generated, even you set TestRecTo(XML).

 

There are 3 ways to define test record options for where/what/how to record test results. This module obeys the following order to define:

1)      Defined by module subroutines: TestRecTo(), SetTestRecPath(), TestRecOpt();

2)      If no definition by subroutines in the scripts, the module will find corresponding environment variables to set record options;

3)      If no definition as described in 1) and 2). This module will use default options.

RUNTIME CONTROL               ^Top

Besides the scripts designer or runner can set the initial runtime controlling parameter for a test task, the test runner can also really runtime control the test. At any time while the test is executing, the test runner can press CTRL-C to pause the test. Once the TestRecorder capture a CTRL-C, the test will be hung up and prompt the tester to select how to do next. The following options will be displayed for selection:

(0)KeepRunning; (1)StopOnStepFail; (2)StopOnCaseFail; (3)PromptOnFail; (4)PromptByStep; (5)PromptByCase; (6)Quit; (D)Debug 

The tester can

1)      Press Enter to continue the test;

2)      Enter a number which indicates a new runtime option for the next steps and cases, and press Enter to continue;

3)      Press ¡®6¡¯ to stop the test.

4)      Press D to debug.

 

Debug  the test           ^Top

When in the Debug mode, the test is hung up, the tester can:

1)      Enter commands to execute, the objects/variables/functions in main name space can be used; In addition, you can input multi-line command by using a ¡®\¡¯ at the end of each line which has a following line.

2)      Enter Q to return.

 

The debug function let user to get information, set values for some variables or objects and do some necessary operations.
